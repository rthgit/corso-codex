Introduzione: Cos'è OpenAI Codex nel 2026

OpenAI Codex è la piattaforma agentica di coding di OpenAI. Non è un semplice completamento di codice: è un sistema multi-superficie che può leggere, scrivere, eseguire ed effettuare il debug del codice in modo autonomo, in parallelo, su cloud e in locale. Il repository ufficiale è github.com/openai/codex.

Lanciato in preview ad aprile 2025 e ora disponibile su tutti i piani ChatGPT paid (Plus, Pro, Team, Enterprise), Codex è alimentato dalla famiglia di modelli codex-1 (ottimizzata per SWE-tasks) e dai modelli GPT-5.x general-purpose.

ℹ
Cos'è cambiato nel 2026: OpenAI ha progressivamente unificato il branding "Codex". I modelli dedicati (es. code-davinci-002) sono stati deprecati. Oggi Codex CLI usa per default codex-mini-latest (alias di o4-mini) che eccelle nel ragionamento su codice a costo contenuto. Per task architetturali si usa gpt-5.5.

Codex CLI — Il Terminale Potenziato

Installazione, configurazione, comandi avanzati, exec non-interattivo, scripting e context management

1.1 Installazione e Setup

Il Codex CLI è open source, scritto in Rust per garantire latenze sub-millisecondo nella gestione del file system, ed è disponibile su macOS, Windows e Linux. Integra un tokenizer locale per il calcolo esatto del contesto in tempo reale.

# Installazione via npm (richiede Node 18+)
npm install -g @openai/codex

# Primo avvio e autenticazione interattiva
codex

# Autenticazione headless (per CI/CD)
export OPENAI_API_KEY="sk-..."
codex login --headless

# Verifica stato, modello attivo e token rimanenti
codex status

1.2 I 3 Approval Mode Ufficiali — La Scelta Più Importante

OpenAI documenta ufficialmente tre livelli di autonomia per il Codex CLI. La scelta errata causa sia blocchi continui (suggest mode su pipeline CI) sia rischi di sicurezza (full-auto su repository di produzione).

ModeSandboxApproval PolicyQuando Usarlo
Suggest (default)read-onlyCodex mostra le modifiche proposte, tu approvi ogni singola azioneEsplorazione, code review, primo utilizzo su un repository sconosciuto
Auto Editworkspace-writeModifica file autonomamente, ma chiede conferma per comandi shellSviluppo attivo quotidiano. Il compromesso ottimale per la maggior parte dei dev
Full Autoworkspace-writeEsegue tutto in autonomia (on-request): codice + shell + comandiCI/CD, container Docker isolati, pipeline notturne. MAI su repository di produzione senza git checkpoint
 
# Avvio con approval mode esplicito
codex --approval-policy suggest        # Modalità sicura (default)
codex --approval-policy auto-edit      # Modifica file, chiede per shell
codex --full-auto                      # Alias per full auto (su CI/CD)

# Equivalente configurato permanentemente in config.toml
[core]
approval_policy = "auto-edit"

1.3 Modalità Non-Interattiva — exec

Trasforma Codex in uno strumento CLI automatizzato, ideale per pre-commit hooks, pipeline CI/CD o cron jobs. Il flag --max-budget-usd è l'unico guardrail contro loop infiniti o consumi anomali.

# Esecuzione con budget cap (previene billing a sorpresa)
codex exec --max-budget-usd 0.50 "Genera test unitari per src/auth.ts"

# Output JSON per parsing strutturato in bash
codex exec --json "Analizza il codice e restituisci bug" | jq '.bugs[]'

# Full auto su cloud container (sicuro perché isolato)
codex cloud exec --env ENV_ID --full-auto --attempts 3 "Risolvi il bug #123"

# Pipeline pre-commit hook (esempio .husky/pre-commit)
codex exec --approval-policy auto-edit --max-budget-usd 0.10 "Fixa i lint errors"

1.3 Gestione del Contesto (Context Management)

La "Context Window" di Codex non è infinita. Un contesto "inquinato" porta a codice difettoso. I developer senior usano questi comandi per mantenere il contesto "fresco".

Modulo02

AGENTS.md — Istruzioni Persistenti per il Progetto

Governance, convenzioni, comandi di test e "Gotchas"

AGENTS.md (noto anche come CLAUDE.md o regole di sistema) è un file testuale, simile a README.md, che Codex legge automaticamente all'inizio di ogni sessione nel tuo repository.

ℹ
Best Practice 2026: Mantieni AGENTS.md sotto le 200 righe. Usa percorsi espliciti e regole di "Context Freshness" per evitare che l'agente allucini pattern non esistenti.

2.1 Come Codex Scopre AGENTS.md — Il Sistema Gerarchico Ufficiale

Codex non legge un solo AGENTS.md: costruisce una catena di istruzioni gerarchica concatenando più file, dalla root del repo fino alla tua directory corrente. Le regole più vicine alla CWD hanno priorità sulle regole globali.

Ordine di ScopertaFile ControllatoComportamento
1. Globale$CODEX_HOME/AGENTS.override.mdSe esiste, sovrascrive tutto. File di policy aziendale non modificabile dai dev.
2. Globale fallback$CODEX_HOME/AGENTS.mdRegole globali personali (es. stile preferito, tool di default).
3. Root progetto[git-root]/AGENTS.mdRegole standard del progetto, lette da tutto il team.
4. SubdirectoryOgni AGENTS.md nei path intermediRegole per sotto-moduli. Concatenati in ordine root→CWD.
5. CWD[cwd]/AGENTS.md o AGENTS.override.mdMassima priorità. Override locale per test o sviluppo isolato.
ℹ
Limite totale: Il processo si ferma quando la somma dei file raggiunge project_doc_max_bytes (default 32 KiB, configurabile in config.toml). File non inclusi vengono ignorati silenziosamente.

2.2 Configurazione Avanzata di AGENTS.md in config.toml

# ~/.codex/config.toml

# Aumenta il limite se AGENTS.md viene troncato (verificare con `codex status`)
project_doc_max_bytes = 65536  # Default: 32768 (32 KiB)

# Nomi file alternativi se AGENTS.md non esiste (monorepo con naming custom)
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md", "CODEX.md"]

2.3 AGENTS.md Completo per Progetto Node.js/TypeScript

# AGENTS.md

## Progetto
Questa è una API REST in Node.js + TypeScript (Express).

## Architettura e Naming
- src/routes/ → endpoint Express. Usa sempre `kebab-case` per i file di route.
- src/services/ → business logic. Usa classi con dependency injection.
- src/models/ → Prisma schemas.

## Comandi Essenziali
- Build: `npm run build`
- Test: `npm test` (NON modificare i file di configurazione Jest)
- Lint: `npm run lint -- --fix`

## Regole di Sicurezza
- NON salvare mai le chiavi API nel codice. Usa dotenv e `process.env`.
- Verifica sempre il permesso dell'utente nei controller, non nei service.

## "Gotchas" (Diario delle Decisioni)
- [Bug Frequente]: Quando aggiungi un nuovo modello Prisma, ricordati di aggiornare il seed file `prisma/seed.ts`.
- [Tip]: L'autenticazione è gestita dal middleware `authMw`, non applicarlo manualmente alle rotte pubbliche.

## Context Freshness
Prima di iniziare modifiche, esegui sempre `git status` e rileggi l'ultimo PRD.

Model Context Protocol (MCP)

Connettere Codex a GitHub, Linear, Slack, DB e lazy loading

Il Model Context Protocol è lo standard de-facto del 2026 per connettere Codex a fonti dati esterne (Notion, Figma, database SQL) senza copiare/incollare testo.

⚡
Lazy Loading: Gli MCP servers inseriscono molte informazioni nella Context Window di Codex. Usa ENABLE_TOOL_SEARCH="true" per caricare gli strumenti MCP dinamicamente, risparmiando il 30% dei token.
# Configurazione di connessione in ~/.codex/config.toml
[[mcp_servers]]
name = "github"
command = "npx"
args = ["-y", "@modelcontextprotocol/server-github"]
env = { GITHUB_PERSONAL_ACCESS_TOKEN = "$GITHUB_PAT" }

[[mcp_servers]]
name = "postgres"
command = "npx"
args = ["-y", "@modelcontextprotocol/server-postgres", "postgresql://user:pass@localhost/db"]

Una volta configurato, puoi chiedere a Codex: "Leggi la issue #42 su GitHub, interroga il database postgres per trovare l'utente incriminato, e implementa il fix."

Modulo06

Automations — codex-action e CI/CD

GitHub Actions con openai/codex-action, schedule, Triage Inbox e pipeline complete

OpenAI pubblica ufficialmente openai/codex-action (github.com/openai/codex-action): una GitHub Action che esegue Codex all'interno dei tuoi workflow CI/CD. Nessun server da gestire, nessun cron esterno.

⚡
Setup rapido: Vai in Settings → Secrets and variables → Actions del tuo repo e aggiungi OPENAI_API_KEY come secret. Poi copia il YAML qui sotto.

Workflow 1 — PR Review Automatica su ogni Pull Request

# .github/workflows/codex-pr-review.yml
name: Codex PR Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  codex-review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: refs/pull/${{ github.event.pull_request.number }}/merge

      - name: Run Codex Review
        id: codex
        uses: openai/codex-action@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          prompt: |
            Sei un Staff Engineer senior. Analizza le modifiche di questa PR.
            Cerca: bug logici, problemi di sicurezza (injection, auth bypass),
            mancanza di error handling, assunzioni fragili.
            Output formato: CRITICAL / WARNING / SUGGESTION / OK.
          sandbox: read-only

      - name: Commenta sulla PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🤖 Codex Review\n\n${{ steps.codex.outputs.final-message }}`
            })

Workflow 2 — Security Scan Giornaliero con Auto-Fix

# .github/workflows/codex-security-scan.yml
name: Security Scan + Auto-Fix

on:
  schedule:
    - cron: '0 6 * * *'  # Ogni giorno alle 06:00 UTC
  workflow_dispatch:     # Permettere esecuzione manuale

jobs:
  security-fix:
    runs-on: ubuntu-latest
    permissions:
      contents: write
      pull-requests: write
    steps:
      - uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '20'

      - run: npm ci

      - name: Codex Security Fix
        uses: openai/codex-action@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          prompt: |
            1. Esegui `npm audit` e leggi l’output
            2. Per ogni CVE critica o alta, aggiorna la dipendenza
            3. Esegui i test: `npm test`
            4. Se i test passano, crea un branch fix/security-$(date)
               con le modifiche e apri una PR
            5. Se i test falliscono, documenta in SECURITY-REPORT.md
               i fix non applicabili e il motivo
          sandbox: workspace-write

Pattern: Test Time Compute (Reviewer Avversariale)

Una context window fresca trova errori che chi ha scritto il codice non vede. È il pattern più efficace per aumentare la qualità senza costo umano aggiuntivo.

🎯 PROMPT — Reviewer Avversariale con Subagente Isolato
FASE 1 (Agente Principale): Implementa la feature X. Al termine, scrivi la git diff in /tmp/diff-for-review.patch. FASE 2 (Subagente con contesto vuoto): Leggi solo il file /tmp/diff-for-review.patch. Non hai contesto di come è stato scritto. Agisci come Staff Engineer con 15 anni di esperienza. Analizza criticamente: - Memory leak o resource leak - Race conditions - Input non sanitizzato (injection) - Logica difettosa agli edge case - Test mancanti o insufficienti Output obbligatorio: CRITICAL: [problema grave che blocca il merge] WARNING: [problema minore ma da correggere prima del merge] SUGGESTION: [miglioramento opzionale] OK: [aspetti corretti] FASE 3 (Agente Principale): Correggi tutti i CRITICAL prima di rispondermi.

Modulo10

Sandbox, Rules e Sicurezza OS-Level

Architettura sandbox reale (Seatbelt/Landlock/seccomp), Rules e danger-full-access

Architettura Sandbox a Livello di Kernel (Dati Ufficiali OpenAI)

La sandbox di Codex non è un semplice controllo software: è un isolamento a livello di kernel del sistema operativo. Anche se il modello AI venisse compromesso tramite prompt injection, non potrebbe sfuggire ai vincoli del kernel.

OSTecnologiaCosa Blocca
macOSApple Seatbelt (sandbox-exec)Genera dinamicamente profili sandbox in base alla policy selezionata. Blocca accesso al filesystem fuori dal workspace e tutte le chiamate di rete.
LinuxLandlock + seccompLandlock: access control per path a livello kernel. seccomp: filtra le system call, bloccando operazioni di rete e accesso ai file sensibili.
WindowsRestricted User Tokens + Job ObjectsOgni processo figlio gira con token a privilegi ridotti in un Job Object isolato. Limita lettura/scrittura del registry e azioni di sistema.
⚠
Network disabilitata per default: In tutti i modi sandbox, l'accesso di rete è bloccato a livello kernel. Questo previene data exfiltration, C2 e download di dipendenze non autorizzate durante l'esecuzione. Per abilitarlo (es. tool che devono fare fetch) devi usare danger-full-access o configurare eccezioni specifiche.

I 4 Livelli di Sandbox — Tabella Completa Ufficiale

ModoFilesystemNetworkQuando Usarlo

workspace-writeLettura ovunque, scrittura solo nella dir git corrente✗ BloccataSviluppo standard quotidiano. Default consigliato.
writable-rootScrittura su path espliciti definiti dall'utente✗ BloccataMonorepo dove Codex deve scrivere in più directory

Rules — Allowlist e Denylist Deterministiche

Le Rules sono valutate prima del modello: impossibile bypassarle via prompt injection. Si combinano con la sandbox: la sandbox definisce il confine fisico, le Rules definiscono i comandi permessi dentro quel confine.

Codex Enterprise — Governance e Sicurezza

Admin setup, managed config, permission profiles e audit trail

In contesti aziendali, Codex deve operare dentro confini precisi e tracciabili. Il livello Enterprise aggiunge governance centralizzata.

Pillars della Sicurezza Enterprise

PillarImplementazione
Config ManagedUn file config.toml centralizzato distribuito via Git o secret manager aziendale (es. AWS SSM). Gli sviluppatori non possono sovrascrivere le policy di deny.
Permission ProfilesProfili predefiniti per ruolo (es. junior-dev: solo lettura e test; senior-dev: scrittura con approvazione; ci-bot: exec non-interattivo con budget cap).
AGENTS.md GlobaleUn AGENTS.md nella root aziendale (letto da tutti i repo via Git submodule) impone standard di codice uniformi e proibisce pattern non sicuri.
Audit LoggingOgni tool call, file modificato e comando eseguito viene salvato in ~/.codex/log/ con timestamp, costo in token e hash dell'output. Integrabile con SIEM aziendali.
Zero-Trust SandboxModalità workspace-read: Codex può leggere ma non modificare file al di fuori delle directory esplicitamente permesse. Ideale per agent di code review.

Modulo12

Prompt Engineering Avanzato per Codex

Framework RIPER, 5 Principi, self-healing e tecniche master

I 5 Principi del Prompt Codex

  • Contesto del task: cosa si deve fare, perché e qual è la metrica di successo misurabile.
  • Evidenza verificabile: specifica i test da eseguire, i file che devono essere creati o i comandi il cui output prova il successo.
  • Constraint espliciti: di' esplicitamente cosa NON toccare (es. "Non modificare il database schema").
  • Output format: risultato strutturato (JSON, Markdown, diff) per pipeline automatizzate.
  • Iterazione: task incrementali e verificabili a ogni step, non un monolite da validare alla fine.

Framework RIPER (Research → Innovate → Plan → Execute → Review)

Il workflow strutturato definitivo per task ad alta complessità. Impone separazione netta tra fasi: Codex non può "saltare" all'implementazione prima di aver capito il problema.

FaseIstruzione a CodexOutput Atteso
Research"Leggi il codebase. Non scrivere niente. Rispondimi solo con cosa hai capito."Mappa del codice, dipendenze, pattern esistenti.
Innovate"Proponi 3 approcci architetturali diversi per questo problema con pro e contro."Opzioni architetturali, nessun codice.
Plan"Scrivi un piano step-by-step. Per ogni step indica il file, il test e il criterio di successo."Piano dettagliato approvabile.
Execute"Implementa il piano approvato. Un step alla volta. Fermati dopo ogni step e aspetta la mia conferma."Diff incrementali, test passanti.
Review"Usa un subagente con contesto vuoto per fare review avversariale dell'implementazione."Report CRITICAL/WARNING/SUGGESTION.
🎯 PROMPT — Template — Feature Development (Completo)
FEATURE: [Nome della feature] CONTESTO: [Descrizione del progetto e file AGENTS.md rilevanti] REQUISITI FUNZIONALI: - [Requisito 1] — verificabile con: [test specifico] - [Requisito 2] — verificabile con: [test specifico] CONSTRAINT: - NON modificare: [lista file o moduli sensibili] - NON installare nuovi pacchetti senza approvazione DEFINIZIONE DI DONE: - [ ] CHANGELOG.md aggiornato

Modulo Bonus — Trucchi Reali dalla Community

Questa sezione raccoglie pattern pratici emersi dall'uso reale di Codex nel 2026, da repository GitHub, discussioni su Reddit (r/ChatGPTCoding, r/LocalLLaMA) e community Discord di dev AI. Non sono “prompt magici”: sono abitudini operative che migliorano affidabilità, qualità e sicurezza.

ℹ
Regola d'oro 2026: Codex performa al massimo quando lavori per pipeline strutturate, non per comandi isolati. Pianifica, riduci lo scope, implementa in piccoli step, verifica con test e fai review incrociata.

1. Prima pianificazione, poi codice

L'errore più comune è chiedere subito a Codex di modificare file. In progetti reali il risultato migliora drasticamente quando prima chiedi di ragionare su architettura, rischi, tradeoff e piano operativo. Il risparmio di tempo ripaga sempre il "costo" della pianificazione.

🎯 PROMPT — Plan First (Esplorazione Prima del Codice)
Prima non scrivere codice. Esplora silenziosamente. Analizza e rispondimi: 1. Architettura attuale (quali file/moduli sono coinvolti) 2. Obiettivo preciso della modifica 3. Rischi tecnici e dipendenze fragili 4. File che verranno modificati e file che NON devono essere toccati 5. Piano step-by-step con test per ogni step 6. Stima della complessità: bassa / media / alta Solo dopo la mia approvazione del piano, implementa.

Auto-validazione del piano

  • Il piano cita file concreti con path reale?
  • Identifica rischi specifici (non generici)?
  • Propone test verificabili con un comando eseguibile?
  • Distingue tra ciò che Codex sa per certo e ciò che ipotizza?

2. Trasforma richieste vaghe in prompt operativi

Un trucco molto utile è usare prima un modello conversazionale per trasformare un’idea vaga in una specifica precisa da dare a Codex. Il prompt finale deve contenere contesto, vincoli, file, output atteso e definition of done.

🎯 PROMPT — Prompt Builder per Codex
Trasforma questa richiesta vaga in un prompt operativo per Codex. Richiesta: [incolla richiesta] Il prompt finale deve includere: - contesto tecnico - obiettivo - file o directory coinvolte - vincoli - cosa non modificare - test da eseguire - definition of done - formato dell’output

3. Task piccoli battono task enormi

Codex può gestire task complessi, ma in produzione conviene spezzare il lavoro in unità piccole. Ogni step deve produrre un risultato verificabile: un test, un file, una diff, un report o una PR.

Task deboleTask professionale
“Migliora il backend”“Analizza solo src/services, trova duplicazioni, proponi piano, poi refactora un servizio alla volta.”
“Sistema i test”“Esegui i test, classifica i fallimenti, fixa solo i test deterministici, segnala i flaky.”
“Rendi tutto sicuro”“Fai audit solo su auth e input validation. Output: CRITICAL / WARNING / SUGGESTION.”

4. AGENTS.md deve prevenire contesto stantio

Nei workflow con più agenti o modifiche manuali, Codex può lavorare su una fotografia vecchia del repository. Per evitarlo, inserisci regole esplicite nel file AGENTS.md. 

## Context Freshness

Before editing:
- run git status
- re-read every file you are about to modify
- assume another human or agent may have changed the repo
- never summarize stale state
- before final answer, check git diff and tests

Auto-validazione

  • Codex controlla git status prima di modificare?
  • Rilegge i file prima di editarli?
  • Mostra diff finale?
  • Dichiara cosa ha verificato davvero?

5. Dai path espliciti quando il progetto è grande

Anche se Codex può esplorare la codebase, nei repository grandi conviene indirizzarlo. I path riducono ambiguità, token sprecati e modifiche indesiderate.

🎯 PROMPT — Refactor con Path Vincolati
Lavora solo su questi file: - src/services/payment.service.ts - src/services/payment.types.ts - tests/payment.service.test.ts Obiettivo: refactor della logica di pagamento mantenendo la stessa API pubblica. Non modificare: - database schema - API routes - config di deploy Verifica: - npm test -- payment.service.test.ts - npm run lint

6. Usa review avversariale

Un pattern avanzato è separare chi implementa da chi revisiona. Anche con un solo Codex, puoi usare due passaggi: prima implementazione, poi review avversariale della diff.

🎯 PROMPT — Reviewer Avversariale
Agisci come reviewer avversariale. Non implementare codice. Analizza la diff e trova: - bug logici - assunzioni fragili - edge case mancanti - test insufficienti - rischi di sicurezza - regressioni possibili Output: CRITICAL: WARNING: SUGGESTION: TEST MANCANTI: DOMANDE APERTE:

7. Usa agenti multipli solo quando serve davvero

Subagents e worktrees sono potenti, ma non vanno usati per ogni cosa. Sono ideali quando vuoi confrontare approcci diversi o dividere lavoro realmente indipendente.

ScenarioUso consigliato
Bug piccoloUn solo agente, task breve, test regressione.
Refactor rischiosoTre worktrees con strategie alternative.
Feature complessaArchitect → Developer → Tester → Reviewer.
Audit sicurezzaImplementer separato da reviewer avversariale.

8. Sicurezza: mai fidarsi del full-access

Codex può essere estremamente produttivo, ma un agente con permessi ampi può anche fare danni. La community tende a convergere su una regola semplice: permessi minimi, allowlist, review prima di push o deploy. 

# Regole pratiche di sicurezza

- Mai full-access su repository non fidati
- Mai token con permessi eccessivi
- Mai automazioni senza allowlist
- Mai push/deploy senza review
- Non esporre .env, secret o token nei prompt
- Usa sandbox workspace-write come default
- Blocca modifiche a CI/CD e file sensibili con hooks
🎯 PROMPT — Audit Sicurezza Prima di Automazione
Prima di creare questa automazione, fai un audit di sicurezza. Valuta: 1. Quali file può modificare 2. Quali comandi può eseguire 3. Quali secret potrebbe leggere 4. Quali danni può causare 5. Quali allowlist servono 6. Quali hook di protezione aggiungere Output: - rischio basso / medio / alto - mitigazioni obbligatorie - configurazione consigliata

9. Pattern operativo finale: P-S-I-V-R

Per lavorare in modo professionale, usa sempre questo ciclo. Non saltare mai fasi: ogni fase è una rete di sicurezza per quella successiva.

FaseDescrizioneOutput verificabileFallback se fallisce
PlanAnalizza architettura, rischi e file coinvoltiPiano tecnico approvatoChiedi a un LLM diverso di trovare i buchi nel piano
ScopeRiduci il perimetro al minimo necessarioLista file e vincoli esplicitiSpezza ulteriormente il task
ImplementModifica in step piccoli con checkpoint gitDiff incrementale commitTatagit stash e ricomincia da zero
VerifyEsegui test specifici, lint e buildOutput test verdi, zero warning criticiSelf-healing loop (max 5 cicli)
ReviewReview avversariale con subagente isolatoReport CRITICAL/WARNING/SUGGESTIONCorreggi solo i CRITICAL prima di mergiare

10. Handoff Document — Contesto tra sessioni

Quando hai una sessione lunga e devi chiuderla o il contesto è al limite, crea un Handoff Document prima di uscire. La prossima sessione leggerà solo questo file e riprenderà esattamente da dove eri.

🎯 PROMPT — Genera Handoff Document
Prima di chiudere questa sessione, scrivi un file HANDOFF.md con: 1. Cosa stavamo costruendo (obiettivo preciso) 2. Cosa hai già implementato (con percorsi file) 3. Cosa ha funzionato e cosa no (inclusi tentativi falliti) 4. Lo stato attuale dei test (passano? falliscono? quali?) 5. I prossimi 3 step esatti da eseguire nella prossima sessione 6. Eventuali "gotchas" scoperti da non dimenticare Il prossimo agente leggerà solo questo file. Sii chirurgico e preciso.

11. Gestione del Budget di Contesto

Il contesto è la risorsa più preziosa. Monitorarlo attivamente previene il "context rot" (degrado silenzioso della qualità).

Utilizzo ContestoAzione Consigliata
0–50%Lavora liberamente. Nessuna precauzione necessaria.
50–70%Evita di includere file grandi non necessari. Usa @file solo per file specifici.
70–90%Usa /compact focus on [argomento critico] al prossimo stopping point.
90%+/clear obbligatorio o nuova sessione con Handoff Document.

Cheat Sheet — Riferimento Rapido Completo

CLI — Comandi Essenziali

npm install -g @openai/codex             # Installazione (richiede Node 18+)
codex                                     # Avvio sessione interattiva (Suggest mode)
codex --model codex-mini-latest "Prompt"  # Modello leggero per task semplici
codex --model codex-1 "Prompt"            # Modello avanzato per task complessi
codex --approval-policy suggest           # Mostra modifiche, tu approvi tutto
codex --approval-policy auto-edit         # Modifica file, chiede solo per shell
codex --full-auto                         # Tutto automatico (solo su CI/sandbox)
codex exec "Task"                         # Esecuzione non-interattiva
codex exec --json "Task" | jq             # Output JSON parsabile
codex exec --max-budget-usd 0.50 "Task"   # Con budget cap di sicurezza
codex --worktree nome-feature "Task"      # Agente su Git worktree isolato
codex mcp list                            # Lista server MCP connessi
codex status                              # Modello attivo, token usati, costo

Slash Commands in Sessione

/model codex-mini-latest     # Cambia a modello economico
/model codex-1               # Passa a modello avanzato
/compact focus on auth        # Compatta preservando contesto sull'argomento
/clear                        # Azzera contesto (nuovo task non correlato)
/rewind                       # Torna a checkpoint precedente (annulla azioni)
/review                       # Revisione automatica delle diff non committate
/skills                       # Lista skills disponibili nel progetto
/cost                         # Spesa sessione corrente in USD + token
@src/auth/login.ts            # Forza inclusione file nel contesto (token-efficient)
!npm test                     # Esegui shell e inietta output nel contesto

Approval Modes — Tabella di Decisione

suggest     → sandbox: read-only   → approvi tutto: ideale per esplorazione
auto-edit   → sandbox: workspace-write → modifica file, chiede per shell: sviluppo quotidiano
full-auto   → sandbox: workspace-write → tutto automatico: solo CI/CD e container isolati
