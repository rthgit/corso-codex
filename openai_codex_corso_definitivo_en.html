Launched in preview in April 2025 and now available on all paid ChatGPT plans (Plus, Pro, Team, Enterprise), Codex is powered by the codex-1 family of models (optimized for SWE tasks) and the GPT-5.x general-purpose models.

AGENTS.md — Persistent Project Instructions

Governance, conventions, test commands, and "Gotchas"

AGENTS.md (also known as CLAUDE.md or system rules) is a text file, similar to README.md, that Codex reads automatically at the beginning of each session in your repository.

ℹ
2026 Best Practice: Keep AGENTS.md under 200 lines. Use explicit paths and "Context Freshness" rules to prevent the agent from hallucinating non-existent patterns.

2.1 How Codex Discovers AGENTS.md — The Official Hierarchical System

Codex doesn't read just one AGENTS.md: it builds a hierarchical instruction chain by concatenating multiple files, from the repo root down to your current directory. Rules closer to the CWD take priority over global rules.

Discovery OrderChecked FileBehavior
1. Global$CODEX_HOME/AGENTS.override.mdIf it exists, overwrites everything. Company policy file, non-modifiable by devs.
2. Global fallback$CODEX_HOME/AGENTS.mdPersonal global rules (e.g., preferred style, default tools).
3. Project root[git-root]/AGENTS.mdStandard project rules, read by the whole team.
4. SubdirectoriesEvery AGENTS.md in intermediate pathsRules for submodules. Concatenated in root→CWD order.
5. CWD[cwd]/AGENTS.md or AGENTS.override.mdHighest priority. Local override for isolated dev or testing.
ℹ
Total Limit: The process stops when the sum of the files reaches project_doc_max_bytes (default 32 KiB, configurable in config.toml). Files not included are silently ignored.

2.2 Advanced AGENTS.md Configuration in config.toml

# ~/.codex/config.toml

# Increase limit if AGENTS.md gets truncated (check with `codex status`)
project_doc_max_bytes = 65536  # Default: 32768 (32 KiB)

# Alternative filenames if AGENTS.md does not exist (custom naming monorepos)
project_doc_fallback_filenames = ["TEAM_GUIDE.md", ".agents.md", "CODEX.md"]

# .github/workflows/codex-pr-review.yml
name: Codex PR Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  codex-review:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      pull-requests: write
    steps:
      - uses: actions/checkout@v4
        with:
          ref: refs/pull/${{ github.event.pull_request.number }}/merge

      - name: Run Codex Review
        id: codex
        uses: openai/codex-action@v1
        with:
          openai-api-key: ${{ secrets.OPENAI_API_KEY }}
          prompt: |
            You are a senior Staff Engineer. Analyze the changes in this PR.
            Look for: logic bugs, security issues (injection, auth bypass),
            lack of error handling, fragile assumptions.
            Output format: CRITICAL / WARNING / SUGGESTION / OK.
          sandbox: read-only

      - name: Comment on PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: `## 🤖 Codex Review\n\n${{ steps.codex.outputs.final-message }}`
            })

Sandbox, Rules and OS-Level Security

Real sandbox architecture (Seatbelt/Landlock/seccomp), Rules and danger-full-access

Kernel-Level Sandbox Architecture (Official OpenAI Data)

The Codex sandbox is not a simple software check: it is kernel-level isolation of the operating system. Even if the AI model were compromised via prompt injection, it could not escape the kernel constraints.

OSTechnologyWhat it Blocks
macOSApple Seatbelt (sandbox-exec)Dynamically generates sandbox profiles based on the selected policy. Blocks filesystem access outside the workspace and all network calls.
LinuxLandlock + seccompLandlock: kernel-level access control for paths. seccomp: filters system calls, blocking network operations and sensitive file access.
WindowsRestricted User Tokens + Job ObjectsEvery child process runs with reduced-privilege tokens in an isolated Job Object. Limits registry read/write and system actions.
⚠
Network disabled by default: In all sandbox modes, network access is blocked at the kernel level. This prevents data exfiltration, C2, and download of unauthorized dependencies during execution. To enable it (e.g., tools that need to fetch) you must use danger-full-access or configure specific exceptions.

Rules — Deterministic Allowlist and Denylist

Rules are evaluated before the model: impossible to bypass via prompt injection. They combine with the sandbox: the sandbox defines the physical boundary, Rules define the allowed commands within that boundary.

# ~/.codex/config.toml
[rules]

# Allowlist: Codex executes WITHOUT asking for confirmation
[[rules.allow]]
command = "npm"

[[rules.allow]]

[[rules.deny]]

Advanced Prompt Engineering for Codex

RIPER Framework, 5 Principles, self-healing, and master techniques

The 5 Codex Prompt Principles

  • Task context: what needs to be done, why, and what is the measurable success metric.
  • Verifiable evidence: specify the tests to run, the files that must be created, or the commands whose output proves success.
  • Explicit constraints: explicitly state what NOT to touch (e.g., "Do not modify the database schema").
  • Output format: structured result (JSON, Markdown, diff) for automated pipelines.
  • Iteration: incremental and verifiable tasks at each step, not a monolith to validate at the end.

RIPER Framework (Research → Innovate → Plan → Execute → Review)

The definitive structured workflow for highly complex tasks. Enforces a strict separation between phases: Codex cannot "jump" to implementation before understanding the problem.

PhaseInstruction to CodexExpected Output
Research"Read the codebase. Do not write anything. Just reply to me with what you understood."Code map, dependencies, existing patterns.
Innovate"Propose 3 different architectural approaches for this problem with pros and cons."Architectural options, no code.
Plan"Write a step-by-step plan. For each step, indicate the file, the test, and the success criteria."Detailed, approvable plan.
Execute"Implement the approved plan. One step at a time. Stop after each step and wait for my confirmation."Incremental diffs, passing tests.
Review"Use an empty context subagent to perform an adversarial review of the implementation."CRITICAL/WARNING/SUGGESTION report.

Bonus Module — Real Community Tricks

This section collects practical patterns emerging from real Codex use in 2026, from GitHub repositories, Reddit discussions (r/ChatGPTCoding, r/LocalLLaMA) and AI dev Discord communities. These are not "magic prompts": they are operational habits that improve reliability, quality, and security.

ℹ
2026 Golden Rule: Codex performs best when you work for structured pipelines, not isolated commands. Plan, reduce scope, implement in small steps, verify with tests, and cross-review.

1. Plan first, then code

The most common mistake is immediately asking Codex to modify files. In real projects, the result improves drastically when you first ask it to reason about architecture, risks, tradeoffs, and an operational plan. The time saved always repays the "cost" of planning.

🎯 PROMPT — Plan First (Exploration Before Code)
Do not write code yet. Explore silently. Analyze and answer me: 1. Current architecture (which files/modules are involved) 2. Precise goal of the modification 3. Technical risks and fragile dependencies 4. Files that will be modified and files that MUST NOT be touched 5. Step-by-step plan with tests for each step 6. Complexity estimation: low / medium / high Only after my approval of the plan, implement.

Plan auto-validation

  • Does the plan cite concrete files with real paths?
  • Does it identify specific (not generic) risks?
  • Does it propose verifiable tests with an executable command?
  • Does it distinguish between what Codex knows for sure and what it assumes?

2. Turn vague requests into operational prompts

A very useful trick is to first use a conversational model to turn a vague idea into a precise specification to give to Codex. The final prompt must contain context, constraints, files, expected output, and definition of done.

🎯 PROMPT — Prompt Builder for Codex
Turn this vague request into an operational prompt for Codex. Request: [paste request] The final prompt must include: - technical context - goal - involved files or directories - constraints - what not to modify - tests to run - definition of done - output format

3. Small tasks beat huge tasks

Codex can handle complex tasks, but in production, it's better to break the work into small units. Each step must produce a verifiable result: a test, a file, a diff, a report, or a PR.

Orchestrator + Vector Database

Long memory for agents: what to save, who decides, how to retrieve context without burning tokens.

In advanced workflows with multiple agents, using only Markdown files and chat history is not enough. Every worker can produce reports, state, and durable facts, but a higher level is needed to decide what information becomes global memory. That level is the orchestrator.

Fundamental principle: agents produce knowledge, but they must not be the ones to decide what enters global memory. The orchestrator validates, normalizes, and updates the vector database.

Correct architecture

                 │    ORCHESTRATOR    │
                 │  manager / router  │
                 └─────────┬──────────┘

                           │
                 ┌─────────▼──────────┐
                 │   VECTOR DATABASE  │
                 │   global memory    │
                 └─────────┬──────────┘
                           │
               targeted retrieval for task
                           │
      ┌────────────────────┼────────────────────┐
      ▼                    ▼                    ▼
tests-quality         backend-api          frontend-ui
local state           local state          local state
state/report          state/report         state/report

Real Architecture of an Enterprise Plugin

A plugin should not be treated as a folder of prompts. It is an internal product: it has a mission, explicit boundaries, versioned instructions, sample inputs, failure cases, security policy, and a repeatable validation method. If a skill only says "review this", results will vary. If it defines scope, priorities, severity, and output format, Codex becomes much more stable and easier to integrate into team workflows.

ElementDesign QuestionCommon Mistake
ManifestHow is the plugin discovered, described, and authorized?Generic name, vague description, no ownership.
SkillWhich procedure should Codex follow without reinventing it every time?Overlong skill, not testable, full of implicit exceptions.
CommandWhich task should be runnable as a repeatable routine?A command that mixes analysis, file edits, tests, and deploy.
MCPWhich external data does the workflow truly need?Connecting everything, increasing risk surface and noise.
AssetsWhich templates, schemas, or examples reduce ambiguity?No canonical example of a successful output.

Testing a Plugin Like Software

The right way to validate a plugin is to run it against small but realistic repositories: a clean repo, a repo with broken tests, a repo with a known vulnerability, a legacy repo without tests, and a repo with incomplete documentation. For every scenario, define the acceptable output in advance. Without verifiable expectations, you are not testing the plugin; you are only reading a plausible answer.
