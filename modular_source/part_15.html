Module14

Orchestrator + Vector Database

Long memory for agents: what to save, who decides, how to retrieve context without burning tokens.

In advanced workflows with multiple agents, using only Markdown files and chat history is not enough. Every worker can produce reports, state, and durable facts, but a higher level is needed to decide what information becomes global memory. That level is the orchestrator.

Fundamental principle: agents produce knowledge, but they must not be the ones to decide what enters global memory. The orchestrator validates, normalizes, and updates the vector database.

Correct architecture

                 │    ORCHESTRATOR    │
                 │  manager / router  │

                 │   VECTOR DATABASE  │

Orchestrator GlobalVector DB / memory indexDurable facts retrievable across tasks and agents.Orchestrator only.

Why not let agents write directly to the vector DB

RiskWhat happensOrchestrator solution
NoiseAgents save irrelevant details.Filters only useful durable facts.
ContradictionsTwo agents save incompatible versions of the same fact.Deduplicates and validates before insertion.
Toxic contextUnverified hypotheses become memory.Requires confidence and verification.
Wasted tokensToo much retrieval, low signal.Top-k targeted by task and role.

Payload that an agent must deliver to the orchestrator

The agent doesn't save directly. It produces a candidate payload. The orchestrator decides what to insert. 

{
  "agent": "tests-quality",
  "task_id": "split-focused-ctest-suites",
  "candidate_memory": [
    {
      "fact": "The monolithic test executable has been split into focused CTest suites.",
      "confidence": "high",
      "scope": "global",
      "tags": ["tests", "ctest", "cmake", "architecture"],
      "evidence": [
        "CMakeLists.txt updated",
        "ctest --test-dir build -C Debug --output-on-failure passed"
      ]
    },
    {
      "fact": "Persistence and CLI tests use unique temp roots to avoid parallel hazards.",
      "confidence": "high",
      "scope": "global",
      "tags": ["tests", "temp-files", "parallelism"],
      "evidence": [
        "tests/TestSupport.hpp updated",
        "ge_tests_persistence passed"
      ]
    }
  ],
  "do_not_store": [
    "Speculation about future UI test strategy not yet approved."
  ],
  "open_questions": [],
  "blocked_by": []
}

Orchestrator rules to save memory

Evaluate future utilityWill the fact help another task or agent?
Verify evidenceIs there a command, diff, or file proving the fact?
Classify scopeAgent local, project global, temporary, or do not store.
NormalizeRemove ambiguity, add tags, link paths and tests.
Insert or updateDeduplicate similar facts and update obsolete ones.

Fake vector DB to start without infrastructure

For the course, you don't need to start with a real vector database right away. You can simulate the concept with a JSON indexed by tag, path, and text. The important thing is teaching the discipline of targeted retrieval. 

{
  "memory_items": [
    {
      "id": "mem_tests_001",
      "text": "The monolithic test executable has been split into focused CTest suites.",
      "tags": ["tests", "ctest", "cmake"],
      "paths": ["tests/", "CMakeLists.txt"],
      "source_agent": "tests-quality",
      "confidence": "high",
      "last_verified": "2026-04-25",
      "evidence": [
        "ctest --test-dir build -C Debug --output-on-failure"
      ]
    }
  ]
}

Targeted retrieval before a new task

When a new task starts, the orchestrator should not load the entire memory. It must retrieve only a few elements relevant to the query, role, and path. 

{
  "retrieval_request": {
    "task": "Add new persistence regression tests",
    "agent": "tests-quality",
    "query": "persistence tests temp roots CTest no_ui",
    "tags": ["tests", "persistence", "ctest"],
    "paths": ["tests/", "CMakeLists.txt"],
    "top_k": 5
  }
}
PROMPT — Orchestrator: update global memory
Act as orchestrator. Input: - agent's state.json - agent's report.md - agent's handoff.md - final diff - verification commands Task: 1. Extract only verified durable facts 2. Discard hypotheses, opinions, and temporary details 3. Classify each fact as local/global/do_not_store 4. Add tags, paths, evidence, and confidence 5. Update the fake vector DB JSON 6. If you find contradictions with existing memory, report a conflict instead of overwriting Output: - memory_update.json - rejected_items with reason - retrieval hints for future tasks
PROMPT — Orchestrator: prepare context for a worker
Act as orchestrator. I need to start the worker: [agent-name] Task: [task description] Before passing it context: 1. Search the global memory only for relevant facts 2. Retrieve a maximum of top_k=5 items 3. Include only facts with medium/high confidence 4. Avoid duplicates 5. Also select role card, local state, and open questions of the worker 6. Prepare a compact brief Output for the worker: - mission - retrieved context - owned paths - constraints - open questions - definition of done

When to use vector DB and when not to

Use vector DBDon't use vector DB
Cross-session memory.Stable behavior rules: use AGENTS.md.
Verified durable facts.Current operational state: use state.json.
Retrieval from many historical reports.Short checklist always valid: keep it in course or role card.
Long multi-agent projects.Small single tasks.
Exercise 14 — Simulate orchestrator + vector DB
  1. Create two workers: tests-quality and backend-api.
  2. Make each produce state.json, report.md and candidate_memory.
  3. Act as orchestrator and save only verified facts in .codex/memory/vector_db.json.
  4. Start a third task and do top_k=5 retrieval from memory.
  5. Verify that the new agent receives useful context without re-reading the entire repo.
Auto-validation:
  • Do agents not write directly to global memory?
  • Does every memory item have evidence and confidence?
  • Is retrieval top-k and targeted to the task?
  • Do AGENTS.md, role card, state.json, and vector DB have different responsibilities?
  • Does global memory contain durable facts, not temporary logs?
  • Does a new agent receive less but more relevant context?
