Module13

Role cards for specialized agents

How to define a serious worker agent: mission, boundaries, local context, escalation, and report format.

The previous module defines the operational state of the agent. This module instead defines the role card: the document that tells an agent who it is, what mission it has, what areas it owns, what it must avoid, and when it must stop to ask the manager.

Fundamental difference:
  • state.json says what the agent has done and verified.
  • The role card says how the agent must behave before, during, and after work.

Why a role card is more advanced than a prompt

Simple promptProfessional role card
"Improve the tests."Defines mission, product goal, boundaries, files to read, risks, escalation, and output.
Does not distinguish root context and local context.Explicitly states which files are worker state and which are managerial context.
Can modify too much.Sets initial read-only mode and future write scope.
Doesn't know when to stop.Has precise escalation rules to the manager.

Real template — tests-quality worker

# Agent: tests-quality

## Mission

Improve the test strategy for the engine and keep no-UI validation reliable.

## Product Goal

Most engine/editor behavior should be testable headlessly, with deterministic regression coverage and no accidental reliance on the ImGui shell.

## Inherited Project Context

This folder is the `tests-quality` worker's local Codex context. Start here, not in the orchestrator's root `.codex` folder.

Read role-local files first:

1. `.codex/AGENTS.md`
2. `.codex/memory.md`
3. `.codex/memory_log.json`
4. `.codex/work_history.md`
5. `.codex/questions.md`

The manager may additionally provide root project context from `j:\game_engine\.codex\AGENTS.md`, `deep-research-report.md`, `important_sites.txt`, and `planning-and-problem-solving-guide.md` when needed. Do not treat those root files as this worker's own state.

## Owned Areas

- Initial work is read-only unless manager approves implementation.
- Proposed future write scope:
  - `tests/`
  - test-only CMake structure.
  - docs covering test strategy.
- Do not refactor production code without manager approval.

## Required Behavior

- Identify monolithic-test risks and propose focused test suites.
- Preserve `build_no_ui` as a valid headless validation path.
- Avoid parallel test hazards caused by shared temp filenames.

## Escalate To Manager If

- Splitting tests requires broad CMake restructuring.
- A production behavior is untestable without adding an API.
- A test reveals a real product behavior ambiguity.

## Report Format

- Summary.
- Current test risks.
- Proposed suite split.
- Immediate high-value tests.
- Commands run.

Advanced pattern: local worker context

The most important part of the example is this: the worker doesn't start from the orchestrator root, but from its own local folder. This allows having separate memory and instructions for each agent. 

.codex/
  agents/
    tests-quality/
      AGENTS.md
      memory.md
      memory_log.json
      work_history.md
      questions.md
      state.json
      report.md
      handoff.md

    backend-api/
      AGENTS.md
      memory.md
      memory_log.json
      work_history.md
      questions.md
      state.json
      report.md
      handoff.md
Mistake to avoid: don't make every worker read the entire `.codex` root as if it were its state. The manager can provide global context, but the worker's operational state must remain local to its role.

How to use it in the Desktop App

Create the workerOpen a dedicated task in the Desktop App with a role name, e.g., tests-quality.
Paste the role cardThe role card becomes the initial brief of the thread.
Enforce read-onlyFirst phase analysis only: no changes until the manager approves.
Request reportThe agent must produce risks, plan, suite split, immediate tests, and commands.
Approve implementationOnly after manager review are modifications authorized on owned_paths.

Exercise 13 — Creating three real workers
  1. Create three role cards: tests-quality, backend-api, frontend-ui.
  2. Assign different owned areas to each.
  3. Define at least three escalation conditions for each worker.
  4. Start three threads in the Desktop App, one per worker.
  5. Each worker must initially work in read-only.
  6. The manager only approves plans, not immediate modifications.
  7. At the end, compare reports and handoffs.
Auto-validation:
  • Does the role card clearly distinguish mission and product goal?
  • Does the worker know which local files to read first?
  • Is the write scope limited?
  • Are escalation conditions concrete?
  • Is the final report sufficient for a human reviewer?
  • Can a new thread continue the work using only role card, state, and report?
