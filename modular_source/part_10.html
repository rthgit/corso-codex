IDE Extension — Editor Pairing

VS Code Agent Mode, automatic documentation and inline review

The Codex extension for VS Code goes beyond classic autocomplete: it introduces an Agent Mode that allows delegating multi-step tasks directly from the editor without switching to the terminal.

# Install the extension from VS Code Marketplace:
code --install-extension openai.codex

# Or update to the latest version
code --install-extension openai.codex --force

Advanced IDE Workflows

Module10

Sandbox, Rules and OS-Level Security

Real sandbox architecture (Seatbelt/Landlock/seccomp), Rules and danger-full-access

Kernel-Level Sandbox Architecture (Official OpenAI Data)

The Codex sandbox is not a simple software check: it is kernel-level isolation of the operating system. Even if the AI model were compromised via prompt injection, it could not escape the kernel constraints.

OSTechnologyWhat it Blocks
macOSApple Seatbelt (sandbox-exec)Dynamically generates sandbox profiles based on the selected policy. Blocks filesystem access outside the workspace and all network calls.
LinuxLandlock + seccompLandlock: kernel-level access control for paths. seccomp: filters system calls, blocking network operations and sensitive file access.
WindowsRestricted User Tokens + Job ObjectsEvery child process runs with reduced-privilege tokens in an isolated Job Object. Limits registry read/write and system actions.
⚠
Network disabled by default: In all sandbox modes, network access is blocked at the kernel level. This prevents data exfiltration, C2, and download of unauthorized dependencies during execution. To enable it (e.g., tools that need to fetch) you must use danger-full-access or configure specific exceptions.

The 4 Sandbox Levels — Official Full Table

ModeFilesystemNetworkWhen to Use It
read-onlyRead-only everywhere✗ BlockedCode review, planning, safe exploration of unknown repos
workspace-writeRead everywhere, write only in current git dir✗ BlockedStandard daily development. Recommended default.
writable-rootWrite on user-defined explicit paths✗ BlockedMonorepos where Codex must write to multiple directories
danger-full-accessNo limits✓ EnabledOnly in already isolated environments (Docker, CI container). Never locally.
 
# In config.toml — additional writable paths (writable-root mode)
[sandbox]
mode = "writable-root"
writable_roots = [
  "/home/user/project-a",
  "/home/user/shared-lib",
  "/tmp/codex-output"
]

Rules — Deterministic Allowlist and Denylist

Rules are evaluated before the model: impossible to bypass via prompt injection. They combine with the sandbox: the sandbox defines the physical boundary, Rules define the allowed commands within that boundary.

# ~/.codex/config.toml
[rules]

# Allowlist: Codex executes WITHOUT asking for confirmation
[[rules.allow]]
command = "npm"
args = ["test", "run", "ci", "audit", "install"]

[[rules.allow]]
command = "git"
args = ["status", "diff", "log", "branch", "add", "commit"]

# Denylist: BLOCKED always, even if explicitly requested by the model
[[rules.deny]]
command = "git"
args = ["push", "--force"]

[[rules.deny]]
pattern = ".*\\.env$"  # Blocks reading/writing of any .env file
